## 1. Background: Why Maker Over Taker

### The Meta Shift (Jan–Feb 2026)

Two things happened quietly in early 2026 that wiped out the old playbook:

**January 2026 — Polymarket introduced dynamic taker fees:**
```
Fee = C × 0.25 × (p × (1 - p))²
```
- Maximum fee: ~1.56% at p=0.50 (exactly where latency arb was most profitable)
- Fee approaches 0% near the extremes (p→0 or p→1)
- Taker fees fund a daily USDC maker rebate pool

**February 18, 2026 — Polymarket silently removed the 500ms taker delay:**
- Previously, all taker orders waited 500ms before executing
- Market makers used that window to cancel stale quotes (free insurance)
- Now: taker orders execute immediately, no cancellation window

**Combined effect:** The arbitrage bot that made $515K/month exploiting Binance→Polymarket price lag? Dead. The fee alone exceeds the spread it was capturing.

### The New Meta in One Line

> **Be a maker, not a taker.**

Makers pay zero fees. Makers receive daily USDC rebates. Top bots are now profitable on rebates alone before even capturing spread.

## 2. How Polymarket's CLOB Works

Polymarket runs a **hybrid-decentralized Central Limit Order Book (CLOB)**:
- Order matching happens **off-chain** (fast, zero gas per match)
- Settlement happens **on-chain** on Polygon (non-custodial)
- Orders are EIP-712 signed typed structured data

**Binary market structure:**
```
YES token + NO token = $1.00 always (at settlement)
```
- If YES wins → YES token → $1.00, NO token → $0.00
- If NO wins → NO token → $1.00, YES token → $0.00

**Key CLOB mechanics:**
- Price priority first, time priority second (standard exchange rules)
- Partial fills are supported — your order can be split across multiple counterparties
- Maker order = limit order that rests in the book waiting for a match
- Taker order = market or aggressive limit that immediately hits a resting order
- Orders are signed with your private key and submitted to the CLOB operator via API

**Market identification:**

Each binary market has:
- A `conditionId` (the market)
- Two `tokenId`s: one for YES, one for NO
- Each tokenId has its own independent order book

**5-minute market slug is fully deterministic:**
```rust
let window_ts = now - (now % 300);  // floor to 5-min boundary
let slug = format!("btc-updown-5m-{}", window_ts);
```
There are exactly 288 BTC 5-min markets per day. You can pre-compute every market's slug and tokenId for the entire day upfront.

## 3. Fee & Rebate Mechanics

### Taker Fee Formula

Applies only to: **5-min crypto markets, 15-min crypto markets, NCAAB, Serie A**

```
Fee = C × 0.25 × (p × (1 - p))²
```

Where `p` = contract price (0.0 to 1.0) and `C` is set by Polymarket.

| Price (p) | Approx Taker Fee |
|-----------|-----------------|
| 0.50      | ~1.56% (maximum) |
| 0.40/0.60 | ~1.13%           |
| 0.30/0.70 | ~0.56%           |
| 0.10/0.90 | ~0.09%           |
| 0.05/0.95 | ~0.02%           |

Trading near 50% as a taker needs a >1.56% edge just to break even. Near 90/10, fees are negligible — this is where late-window directional trades live.

### Maker Rebate Program

- Makers pay **zero fees** on any market
- Daily USDC rebates paid from the taker fee pool
- `your_rebate = (your_filled_maker_volume / total_filled_maker_volume) × rebate_pool`
- Competition is per-market only — you don't compete globally

**Quadratic scoring bonus:** Polymarket's liquidity scoring exponentially rewards quotes closer to midpoint. A quote 1¢ from mid earns ~4× more than a quote 2¢ from mid. Tight quoting is disproportionately rewarded.

### Critical: `feeRateBps` in Order Signatures

Every order signed for fee-enabled markets **must include** `feeRateBps` in the payload:

```json
{
  "salt": "...",
  "maker": "0x...",
  "signer": "0x...",
  "taker": "0x...",
  "tokenId": "...",
  "makerAmount": "50000000",
  "takerAmount": "100000000",
  "feeRateBps": "150"
}
```

If the field is missing or the value doesn't match the current fee rate from `GET /fee-rate?tokenID={id}`, the order is **silently rejected**. Always query the fee rate fresh before signing. Never hardcode it.

## 4. Strategy 1 — Continuous Bid-Ask Spread Market Making

### How It Works

You post **resting limit orders on both the YES and NO sides** simultaneously throughout the market window. When both sides get filled, you capture the spread — profit independent of the outcome.

**The core math:**

```
You post:  YES BUY at $0.48,  NO BUY at $0.48

Someone fills your YES at $0.48 → you hold YES
Someone fills your NO at $0.48  → you hold NO

At settlement:
  YES wins: YES resolves at $1.00, NO at $0.00 → collected $1.00 on $0.96 spent
  NO wins:  NO resolves at $1.00, YES at $0.00 → collected $1.00 on $0.96 spent

Either way: gross profit = $0.04 per pair
Spread capture per round trip: $0.04 / $0.96 = 4.2%
```

You don't care who wins. You're the casino, not the gambler. On top of spread, every filled maker order earns a proportional share of that market's daily rebate pool.

### Quote Pricing — The Stoikov Model (Adapted)

Standard market making uses the Avellaneda-Stoikov model. For binary prediction markets, adapt it:

```
mid  = (best_ask + best_bid) / 2   // or your own fair value estimate
spread = max(min_spread, σ × risk_aversion × inventory_penalty)

bid  = mid - spread/2
ask  = mid + spread/2
```

Key adaptations for Polymarket binaries:
- `σ` = realized volatility of the YES price over recent windows (not raw BTC vol)
- `inventory_penalty` = increases spread width as your YES/NO imbalance grows
- `min_spread` = minimum to beat fees; 2¢ minimum recommended at mid prices

### Inventory Risk: The Core Problem

This is what kills market makers. If you post YES at $0.48 and news breaks pushing YES to $0.90, your $0.48 YES orders get filled instantly (adverse selection) before you can cancel. You now hold YES at $0.48 in a market that's actually 90¢.

**Mitigation techniques:**

1. **Widen spreads during high volatility** — use Binance ATR as a signal; if volatility spikes, widen spread or pause
2. **Inventory skewing** — if you're long YES, shade your YES ask lower and NO bid lower to rebalance
3. **Hard inventory caps** — stop quoting if net directional exposure exceeds X USDC
4. **News event detection** — if BTC moves >0.3% in 30s, cancel all orders and pause
5. **Market selection** — prefer markets with historically low volatility and stable midpoints

### The Cancel/Replace Loop

The core operational loop. Must complete in **<100ms** total:

```
loop:
  1. Receive Polymarket WS orderbook update OR Binance price tick
  2. Recalculate fair value + optimal bid/ask
  3. If quotes are stale (price moved beyond threshold):
     a. Cancel existing YES + NO resting orders (batch cancel preferred)
     b. Compute new quotes with fresh feeRateBps
     c. Sign new orders
     d. Submit new YES + NO orders simultaneously
  4. Sleep until next event
```

Without the 500ms buffer, if your quotes go stale and a taker sees them before you cancel, they pick off your mispriced order. You get filled at a price that's bad for you.

**Target loop timings:**
- Order book update latency: <5ms (WebSocket only)
- Cancel request round-trip: <20ms on co-located VPS
- Recalculate + sign: <5ms (Rust is critical here)
- Submit new orders: <20ms round-trip
- Total budget: <100ms end-to-end

### Position Merging (Critical for Capital Efficiency)

When you hold both YES and NO shares in the same market, they cancel out. Every YES+NO pair = $1.00 redeemable immediately, freeing capital for the next window.

Without merging: 100 YES + 100 NO = $100 locked until settlement  
With merging: $100 USDC freed immediately, redeployed

Automate on-chain merging via the CTF Exchange contract after every successful round-trip fill.

### Market Selection Criteria

| Criterion | What to Look For |
|-----------|-----------------|
| Volatility | Low short-term vol in the YES price |
| Volume | >$5K daily volume (enough takers to fill your quotes) |
| Spread | Current spread >2¢ (room for you to operate) |
| Time to resolution | Avoid markets within 30 min of resolving |
| Liquidity rewards | Check if market has active rewards program |

### Expected Performance

Based on documented bots:
- `@defiance_cr`'s bot: ~$700–800/day at peak (now open-sourced)
- Typical spread per round trip: 2–6¢
- Fill rate: 30–60% (not every quote gets filled — that's fine)
- Monthly returns: 2–8% on deployed capital, highly dependent on market selection

## 5. Strategy 2 — Late-Window Directional Maker

### How It Works

You use Binance spot as a directional signal, but enter **as a maker**, not a taker. The key insight:

> At T-10 to T-30 seconds before a 5-min window closes, BTC price direction is largely locked in. Polymarket's odds haven't fully priced it in yet. You post a maker order on the high-probability side at slightly below fair value — if filled, you win at settlement with known edge.

**Why maker and not taker?**
- Posting a limit at $0.90–$0.95 costs you zero fees
- If filled at $0.92, you profit $0.08 at settlement
- A taker buying the same contract pays up to 0.09% in fees (low near extremes, but still non-zero)
- As a maker, you earn rebates on top

### The Signal: Window Delta

```rust
let window_pct = (current_btc_price - window_open_btc_price) / window_open_btc_price * 100.0;
```

This single number directly answers the market's question. It is the most important signal by far.

| Window Delta | Signal Strength | Action |
|-------------|----------------|--------|
| > +0.10%    | Strong UP      | Post YES maker at $0.88–$0.93 |
| +0.04–0.10% | Moderate UP    | Post YES maker at $0.70–$0.85 |
| -0.04–+0.04% | Ambiguous     | Stay out (coin flip) |
| -0.04 to -0.10% | Moderate DOWN | Post NO maker at $0.70–$0.85 |
| < -0.10%    | Strong DOWN    | Post NO maker at $0.88–$0.93 |

### The Oracle Timing Edge

Settlement uses **Chainlink Data Streams** — a pull-based oracle providing BTC price snapshot at exactly the window end timestamp.

Key timings:
- Chainlink updates every ~10–30 seconds OR on 0.5% price deviation
- There is typically a 2–5 second lag between real exchange price and oracle snapshot
- Polymarket's displayed odds also lag because they're derived from the oracle, not directly from Binance

**The edge:** You're watching Binance WebSocket tick-by-tick. Polymarket's probability display lags your real-time signal by 2–15 seconds. In the final 20 seconds, that's enough to identify a mispriced contract and post a maker order before the market adjusts.

**Validated:** 55–60% win rate on 1,000 BTC 5-min historical windows with simple window delta signal, vs. 50% random. Not huge, but consistent — position sizing and execution quality determine net profitability.

### Entry Timing

```
T = 300s → Window opens. Ignore entirely.
T = 240s → Monitor window_pct. No action.
T = 60s  → Evaluate signal strength. If strong directional, begin preparing.
T = 30s  → If signal confirmed and threshold met: POST MAKER ORDER.
T = 10s  → Last chance. Only if delta is extremely strong (>0.15%).
T = 5s   → Hard cutoff. DO NOT ENTER.
T = 0s   → Settlement. Chainlink snapshot taken.
```

Why no entries after T-5? Polygon transaction confirmation takes 2–5 seconds. If you submit at T-3, the order may not confirm by T-0, resulting in a missed fill.

### Supplementary Signals (Confirming Only)

Use these to confirm the window delta signal, not replace it. More signals = more overfitting.

1. **1-min EMA crossover** — if 5-EMA > 20-EMA on Binance, confirms UP bias
2. **Volume spike** — high volume in the direction of delta strengthens conviction
3. **RSI context** — extreme RSI (>75 or <25 on 1-min) may signal reversal risk; tighten your threshold
4. **Order book imbalance** — if Binance BTC book has 3× more bids than asks, confirms UP

### Position Sizing

```
bet_size = bankroll × kelly_fraction

// Simplified Kelly for binary:
kelly_fraction ≈ win_prob - (1 - win_prob)  // at 55% win rate → ~0.10 Kelly

// Conservative approach:
bet_size = bankroll × 0.01 to 0.03   // 1–3% per trade
```

Do not size more than 3% per trade. 288 markets per day means you will have losing streaks. Even a 60% win rate has runs of 8–10 losses in a row.

### Late-Window Maker Order Mechanics

Post a **limit buy** on the YES or NO token at your target price. Near settlement, the book gets thin. Often the best ask is $0.95+ and you're the only buyer.

**When there are no asks:**  
Post a GTC limit buy at $0.95. You become the liquidity. Anyone wanting to sell YES hits your resting order. If filled, you profit $0.05 at settlement with zero fees and earn a rebate.

**Order type:** Limit buy (maker). NOT a market buy — that's a taker order, you pay fees and likely get a terrible price on a thin book.

### Lock-Free Order Book State

```rust
use dashmap::DashMap;
use rust_decimal::Decimal;
use std::sync::atomic::{AtomicI64, Ordering};

pub struct OrderBook {
    pub yes_bids: DashMap<Decimal, Decimal>,  // price → size
    pub yes_asks: DashMap<Decimal, Decimal>,
    pub no_bids:  DashMap<Decimal, Decimal>,
    pub no_asks:  DashMap<Decimal, Decimal>,
    pub last_update_ms: AtomicI64,
}

impl OrderBook {
    pub fn mid_price(&self) -> Option<Decimal> {
        let best_bid = self.yes_bids.iter()
            .map(|e| *e.key()).max()?;
        let best_ask = self.yes_asks.iter()
            .map(|e| *e.key()).min()?;
        Some((best_bid + best_ask) / Decimal::TWO)
    }
}
```

### Fee Rate Query (Before Every Order)

```rust
#[derive(serde::Deserialize)]
struct FeeRateResponse {
    fee_rate_bps: u64,
}

async fn get_fee_rate(client: &reqwest::Client, token_id: &str) -> anyhow::Result<u64> {
    let url = format!("https://clob.polymarket.com/fee-rate?tokenID={}", token_id);
    let resp: FeeRateResponse = client.get(&url)
        .send().await?
        .json().await?;
    Ok(resp.fee_rate_bps)
}
```

Never cache longer than 60 seconds. Polymarket can update this at any time without notice.

### Inventory Tracker

```rust
use rust_decimal::Decimal;
use rust_decimal_macros::dec;

pub struct InventoryState {
    pub yes_shares: Decimal,
    pub no_shares:  Decimal,
    pub usdc_spent: Decimal,
    pub usdc_collected: Decimal,
}

impl InventoryState {
    pub fn net_directional_exposure(&self) -> Decimal {
        (self.yes_shares - self.no_shares).abs()
    }

    pub fn redeemable_pairs(&self) -> Decimal {
        self.yes_shares.min(self.no_shares)
    }

    pub fn should_pause_quoting(&self, max_exposure: Decimal) -> bool {
        self.net_directional_exposure() > max_exposure
    }

    pub fn realized_pnl(&self) -> Decimal {
        self.usdc_collected - self.usdc_spent
    }
}
```

## 9. Critical Pitfalls & Gotchas

### The 500ms Delay Is Gone — Code Accordingly

Every tutorial, AI-generated snippet, and older open-source bot was written when the 500ms buffer existed. It doesn't anymore. Any code that relies on delayed execution, delayed cancellation windows, or the old taker-friendly timing model **will lose money from trade one.**

When you ask an AI to generate Polymarket bot code, it will write the old version. Check every code suggestion against the post-February-2026 rules.

### Hardcoding `feeRateBps`

```rust
// ❌ Never do this
let fee_rate_bps = 150u64;  // Polymarket can change this without notice

// ✅ Always query fresh
let fee_rate_bps = get_fee_rate(&http_client, &token_id).await?;
```

If your signed order's `feeRateBps` doesn't match the live rate, the order is **silently rejected**. No error message, no fill, just nothing. This is one of the most common failure modes when fees were first introduced.

### REST Polling Instead of WebSocket

```rust
// ❌ 200–500ms per call — useless for real-time quoting
loop {
    let book = client.get_order_book(&token_id).await?;
    tokio::time::sleep(Duration::from_millis(100)).await;
}

// ✅ WebSocket: updates arrive in <5ms as they happen
let mut ws = subscribe_market_channel(&token_id).await?;
while let Some(update) = ws.next().await {
    // process immediately on each event
}
```

### Float Arithmetic on Prices

```rust
// ❌ Never use f64 for prices — binary floating point errors
let spread = 0.52_f64 - 0.48_f64;  // may evaluate to 0.039999999... 

// ✅ Always use rust_decimal
use rust_decimal_macros::dec;
let ask = dec!(0.52);
let bid = dec!(0.48);
let spread = ask - bid;  // exactly 0.04
```

### Not Merging YES/NO Pairs

Every unmerged YES+NO pair is capital sitting idle. If Strategy 1 fills 50 round trips per hour without merging, you have hundreds of USDC locked until settlement. Automate the merge call after every successful round-trip. Port the merge logic from `warproxxx/poly-maker`.

### Market Making Near 50% in Volatile Conditions

Near 50% odds + high volatility = maximum adverse selection (your quotes are most likely filled when they're wrong) + maximum taker fees hitting your book. Either widen spreads aggressively to compensate or avoid quoting near 50% in choppy conditions entirely.

### WebSocket Disconnect Behavior

When the Polymarket WebSocket connection drops, the server automatically cancels all your open orders. Your reconnect handler must:
1. Re-subscribe to market channels
2. Re-query current order book state
3. Re-post all your maker quotes

If you just silently reconnect without re-posting, you have no open orders and earn nothing.

### Ignoring the `tick_size_change` Event

When a market price crosses above 0.96 or below 0.04, Polymarket changes the minimum tick size. If you're posting quotes at the old tick size, orders will be rejected. Subscribe to `tick_size_change` events and adjust your pricing accordingly.

### Backtesting Strategy 2 (Late-Window Directional)

Key questions:
1. What window_delta threshold gives the best precision/recall tradeoff?
2. What's the win rate at each entry timing (T-30, T-20, T-10)?
3. What's the average token price at entry (determines expected profit per trade)?

```rust
// Pseudocode for backtesting loop
for candle in historical_btc_5min_candles {
    let window_open = candle.open;
    let final_price = candle.close;
    let true_outcome = if final_price >= window_open { "UP" } else { "DOWN" };

    // Simulate entry at T-30 using price at that timestamp
    let price_at_t30 = candle.price_at_second(270);
    let delta_at_t30 = (price_at_t30 - window_open) / window_open;

    if delta_at_t30.abs() > threshold {
        let predicted = if delta_at_t30 > 0.0 { "UP" } else { "DOWN" };
        let correct = predicted == true_outcome;
        record(delta_at_t30, correct, entry_price_from_book);
    }
}
// Target: >55% win rate at your threshold
```

## 11. Risk Controls Checklist

Before going live, every bot must implement all of these:

**Hard Stops**
- [ ] Daily loss limit (e.g., -5% of capital → halt all trading, alert via Telegram)
- [ ] Per-trade max size (e.g., never more than 3% bankroll per position)
- [ ] Max inventory cap (e.g., if net YES exposure >$200, stop quoting YES)
- [ ] De-peg killswitch: if Binance + Coinbase prices diverge >0.15%, halt immediately
- [ ] Oracle failure detection: if Chainlink hasn't updated in >60s, halt

**Operational**
- [ ] Paper trading mode (toggle via env var — zero real orders placed)
- [ ] Telegram or Discord alerts on: fills, daily PnL, killswitch triggers, errors
- [ ] Local state synced with on-chain shares every 15s (drift reconciliation)
- [ ] Graceful shutdown on Ctrl+C: cancel all open orders before process exits
- [ ] Auto-merge YES+NO pairs after every successful round-trip fill
- [ ] Reconnect logic for WS disconnects that re-posts all quotes on reconnect

**Monitoring**
- [ ] Latency logging: track cancel/replace loop time per cycle; alert if >150ms
- [ ] Fill rate tracking: monitor % of your quotes that get filled
- [ ] Rebate tracking: verify daily USDC rebates are arriving in your wallet
- [ ] Slippage tracking: compare expected vs actual fill prices
- [ ] Position reconciliation: match local inventory state against on-chain holdings every 15s
